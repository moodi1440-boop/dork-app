## 2. قبل أي تعديل

- عرض الخطة والانتظار لـ "OK" قبل التنفيذ

---

## 3. الرفع للـ GitHub والنشر
- الرفع بعد الموافقة فقط، مع رسالة commit واضحة، بدون force push بدون إذن
- خطوات الرفع الإلزامية بالتسلسل:
  1. اختبار محلي — بدون أخطاء
  2. git commit برسالة واضحة
  3. git push للـ GitHub
  4. شيك PR: "Able to merge" + "All checks passed" + "No conflicts"
  5. اضغط "Merge pull request"
  6. انتظر deployment Vercel
  7. قل "وصل التحديث" فقط بعد تأكيد الـ deployment
- لو في conflict: أحله من GitHub مباشرة
- لا أقول "وصل" إلا لما كل شيء تمام

## 19. قواعد استعلامات قاعدة البيانات (NON-NEGOTIABLE)

**يُمنع منعاً باتاً `select=*`:**
```javascript
// ❌ ممنوع
sb("bookings", "GET", null, "?select=*")
// ✅ صحيح
sb("bookings", "GET", null, "?select=id,salon_id,date,status")
```

**القواعد الإلزامية:**
1. تحديد الأعمدة دائماً — ممنوع `select=*`
2. فلتر WHERE إلزامي لجداول: `bookings`, `messages`, `waiting_list`
3. `limit` إلزامي: Messages=50 | Reviews=20 | Bookings=100-500 | Customers=500
4. RLS مفعّل — أضف `Authorization: Bearer ${sessionToken}` في Headers
5. لا Realtime مفتوح بدون cleanup:
```javascript
return () => channel.unsubscribe()
```
6. Circuit Breaker: توقف بعد 3 فشل متتاليين، انتظر 30 ثانية
7. البيانات الثابتة (مدن، إعدادات) في localStorage — Cache 7 أيام
8. لا صور خام — استخدم Thumbnails/Optimized دائماً

**فحص ذاتي قبل كل commit:**
- [ ] بدون `select=*`؟
- [ ] كل استعلام فيه `limit`؟
- [ ] كل Realtime subscription فيها `unsubscribe()`؟
- [ ] البيانات الثابتة في localStorage؟

## 20. نظافة الكود (Zero-Redundancy)
- عند نقل كود: يُحذف من مكانه القديم فوراً
- عند حذف ميزة: حذف كامل (Functions + Components + Variables + Imports + Routes + JSON keys)
- ممنوع وجود Dead Code أو كود معلق

---

## 21. بروتوكول التكامل البرمجي (Binding)
1. **الملف كاملاً دائماً:** يُمنع تقديم أجزاء من الكود — دائماً الملف كاملاً
2. **حرمة الكود القائم:** لا حذف أو استبدال سطر موجود إلا بموافقة صريحة
3. **حذف الميزة = حذف شامل:** Functions + Components + Variables + Imports + Routes + JSON keys
4. **حظر الإعادة:** ميزة حُذفت بقرار → محظور إعادتها إلا بإذن صريح في نفس الجلسة
5. **حظر git revert:** يُمنع الرجوع لتاريخ git قديم بدون إذن صريح — عند الطريق المسدود: توقف واشرح، لا تتصرف منفرداً
6. **تحقق من الأسطر:** إضافة ميزة → الأسطر تزيد | حذف ميزة → الأسطر تنقص
7. **تعارض منطقي:** توقف فوري وإبلاغ المستخدم — لا تعديل حتى يُحل التعارض

---

## 22. الجراحة البرمجية والمسؤولية
1. **قفل النطاق:** أعمل فقط على ما طُلب — ما لم يُذكر = محظور اللمس
2. **المسؤولية الكاملة:** تعديل A أثّر على B → أنا المسؤول، أُصلحه فوراً بلا مبررات
3. **إعلان قبل البدء:** "سأعدل: [الملف/السطر] — لن أمس: كل ما عدا ذلك"
4. **بروتوكول الشك:** إذا التعديل قد يمس جزءاً آخر → أتوقف وأخبر المستخدم فوراً
5. **تقرير بعد الإنجاز:** ماذا عدّلت + ماذا تركت + أي خطر جانبي

---

## 23. الصدق في الإنجاز وانضباط الرفع
1. **"تم" بعد التحقق البصري فقط** — الظن لا يكفي
2. **إذا التحقق يحتاج اختباراً من المستخدم:** قل صراحةً "التأكد الكامل يحتاج اختباراً من طرفك"
3. **فشلان بنفس النهج = توقف كامل:** شخّص السبب الجذري واقترح نهجاً مختلفاً
4. **ترتيب الرفع الإلزامي:** تحقق ← دمج ← رفع ← رقم الإصدار
5. **رقم الإصدار إلزامي** بعد كل رفع ناجح (مثال: L113)

## 24. قائمة الميزات المحذوفة بقرار (محظور إعادتها بدون إذن صريح)

## 25. أولوية معالجة فجوة RLS المفتوحة (Tier 2) — إنفاذ اتفاق قديم مو نقطة جديدة

**⚠️ هذا البند ليس اتفاقاً جديداً — هو تفعيل/إنفاذ لاتفاق موجود من قبل ولم يُنفَّذ:**
- بند 19.4 أعلاه ("RLS مفعّل — أضف `Authorization: Bearer ${sessionToken}` في Headers") ينص على نفس المبدأ بالضبط، وهو موجود بالملف من قبل 2026-07-14
- قبل إعادة تنظيم الملفات بـ2026-06-30، كان الملف يحتوي قسماً تفصيلياً كاملاً اسمه "القاعدة 4️⃣: RLS Awareness — افترض الحماية مفعلة دائماً" (بأمثلة كود، وتحذير صريح: "RLS هو خط الدفاع الأخير ضد تسرب البيانات، لا تحاول الالتفاف عليه") — اختُصر لسطر واحد وقت إعادة التنظيم، لكن المبدأ بقي متّفَقاً عليه
- **الحقيقة:** `sb()` بـ`App.jsx` لين اليوم (2026-07-14) **ما ترسل `authToken` حقيقي أبداً** (معامل ميت بالكود) — يعني هذا الاتفاق **انتُهك عملياً طول هذي المدة**، رغم كونه مكتوباً ومتفَقاً عليه صراحة. اكتشاف اليوم (Tier 2) هو فعلياً اكتشاف "الاتفاق موجود بس ما تنفّذ"، مو اكتشاف مشكلة جديدة من الصفر

**قرار أحمد (2026-07-14):** مشروع Tier 2 (بناء نظام هوية حقيقي `auth.uid()` للصالونات والعملاء + ربط `sb()` بإرسال `authToken` فعلي + إعادة كتابة سياسات RLS لتحل محل `USING (true)` المفتوحة على `customers`/`bookings`/`reviews`/`promo_codes`) **يُنفَّذ أولاً، قبل أي نقطة أخرى متبقية** (بما فيها #28/#29 استعادة الرمز السري، والأخطاء الصغيرة #18/#30/#31/#32) — يلغي/يتجاوز قرار "الأسهل أولاً" السابق نفس اليوم لخصوص هذي النقطة تحديداً.

## 26. إعادة تسمية العلامة التجارية من DORK إلى DAWRAK

**قرار أحمد (2026-07-15/16):** تغيير الاسم الظاهر بالحروف اللاتينية من "DORK" (يُقرأ بالإنجليزية كصفة سلبية) إلى **"DAWRAK"** — أقرب صوتياً لكلمة "دورك" (دَوْرَك) بلهجة سعودية/خليجية. مرّ القرار بمرحلتين: تجربة أولى باسم "DAWRUK" (L131)، ثم تصحيح لـ"DAWRAK" (L132) بعد نقاش عن دقة النطق.

**النطاق المُنفَّذ بالكامل:**
- هيدر التطبيق (`App.jsx` — 3 حالات: عميل/صالون/زائر) + alt text لشعار الدخول
- شعار الدخول (`public/logo.webp` + `public/logo.png`) — استبدال كامل بتصميم جديد من أحمد، بنفس المسار (لا أثر للملف القديم)

- بادئة أكواد الخصم المولَّدة حديثاً (`DORK-XXXXX` → `DAWRAK-XXXXX` للأكواد **الجديدة فقط** — القديمة تبقى صالحة بصيغتها الأصلية)
- ملفات الترجمة الإنجليزية/التركية/الأردية (نصوص يراها مستخدمون غير عرب)
- 4 ملفات SVG شعار قديمة غير مستخدمة بالكود (حُذفت نهائياً بطلب أحمد، أحدها كان يحمل نص "DORK")
- **كل ملفات التوثيق** (`CLAUDE.md`، `تقرير-أمان-RLS.md`، `النقاط-المتبقية.md`، `الفحص-الشامل.md`، `أمان-التطبيق-وقاعدة-البيانات.md`، هذا الملف، `README.md`، `CONTRIBUTING.md`، `SUPABASE_SETUP.md`، `ملف_الاختبارات.md`، ملفات `docs/`) — بما فيها اسم مشروعَي Supabase أنفسهم ("DORK"/"DORK-TEST" → "DAWRAK"/"DAWRAK-TEST")، بقرار صريح من أحمد "نوحّد الاسم" (لا نُبقي ازدواجية بين اسم قديم بالبنية التحتية واسم جديد بالتطبيق)

## 28. خطة تقسيم `App.jsx` — مشروع GitHub منفصل موازٍ (نفس نمط سيدني→مومباي)

**الخلفية:** 3 محاولات سابقة لتقسيم `App.jsx` (9,346 سطر) فشلت وكسرت التطبيق — آخرها فقدت ميزة الإشعارات لمدة طويلة (راجع نقطة #33 بـ`النقاط-المتبقية.md`). أثناء نقاش 2026-07-16 تبيّن إن السبب الجذري **مو طول الملف نفسه**، بل التحقق المتأخر (اختبار بعد تكديس عدة تعديلات مرة وحدة، بدل بعد كل خطوة).

**القرار (أحمد، 2026-07-16):** التقسيم القادم يصير بـ**مشروع GitHub منفصل تماماً** (نفس منطق ترحيل سيدني→مومباي بقاعدة البيانات) — **لا يُلمس المستودع/التطبيق الحالي إطلاقاً** أثناء البناء. الهدف المعلَن: تنظيم يخلي أي حد يفتح الكود "ما يتعب فيه" — نفس تقسيمات المجلدات المعتادة بتطبيقات React الاحترافية، **مو إعادة بناء من الصفر** (نفس المزايا بالضبط، بس مرتبة بملفات/مجلدات منطقية). لا TypeScript ولا نظام اختبارات آلية ولا تغيير بنية إلا لو طُلب صراحة لاحقاً.

```
  views/               ← الشاشات الكبيرة (HomeView, SalonPage, BookView, RegisterView...)
  drawers/             ← CustomerDrawer, SalonDrawer
  panels/              ← StatsPanel, NotifPanel, PromoPanel, MessagesPanel...
  owner/               ← شاشات صاحب الصالون
  customer/            ← شاشات العميل
  lib/                 ← sb()، الإشعارات، اتصال Supabase (النواة المشتركة — تُنقل آخر شي، أعلى خطورة)
  constants/           ← نصوص الإشعارات، بيانات المواقع
  App.jsx              ← يضل صغير، تنسيق/ربط الشاشات فقط
```

**منهجية التنفيذ (بلا استثناء):**
1. يبدأ بالأجزاء الأصفر خطورتها (أيقونات، ثوابت) أولاً — مستقلة تماماً، خطر شبه معدوم
2. كل نقلة (component/مجموعة صغيرة) وحدها، تُختبر حياً فوراً بعدها — مو تكديس عدة نقلات ثم اختبار بالآخر
3. النواة المشتركة (`sb()`, إدارة الجلسات، الإشعارات) **لا تُنقل إلا آخر شي**، وبعد ما يثبت نجاح كل الباقي
4. **بلا أي وعد "مضمون" أو نسبة نجاح مؤكَّدة** — التزام صريح من كلود بهذا الخصوص (راجع نقطة #33)
5. لا يبدأ التنفيذ الفعلي قبل جلسة جديدة مخصّصة له (**لم يبدأ بعد** اعتباراً من هذا التاريخ)

**⏳ نقطة مفتوحة لم تُحسَم بعد:** كيف تُطبَّق إصلاحات/ميزات عاجلة على التطبيق الحالي أثناء بناء النسخة الجديدة بالتوازي؟ (تطبَّق بالمكانين، أو يجمَّد تطوير القديم مؤقتاً) — يُحسم عند بدء التنفيذ الفعلي.
