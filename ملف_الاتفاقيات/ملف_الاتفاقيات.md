## 🔄 0. الاستراتيجية الهجينة (Hybrid Workflow) - أول اتفاق

### البيئة السحابية (Remote Sandbox) vs البيئة المحلية (Local Environment)

**الاتفاق:** سنعتمد نموذج عمل هجين يتناسب مع موقعك وأدواتك المتاحة:

#### 1️⃣ **عندما تكون خارج البيت (من الآيباد) 📱**
- **إشارة التحويل:** تقول "أنا أعمل الآن من الآيباد خارج البيت"
- **البيئة المستخدمة:** Remote Sandbox (السحابية)
- **آلية العمل:**
  - أقوم بالتعديلات مباشرة في البيئة البعيدة
  - `git commit` بعد كل مجموعة تغييرات
  - `git push -u origin claude/arabic-greeting-DtgO2` فوراً
  - **المزايا:** بدون تثبيت محلي، كل شيء في السحاب، يسهل العمل من أي مكان

#### 2️⃣ **عندما تكون في البيت (من الكمبيوتر) 💻**
- **إشارة التحويل:** تقول "أنا أعمل الآن من جهاز الكمبيوتر في البيت"
- **البيئة المستخدمة:** Local Environment (المحلية)
- **آلية العمل:**
  - تعديل **مباشر** على `/home/user/dork-app` على جهازك المحلي
  - تشغيل `npm run dev` ومعاينة فورية (Hot Reload)
  - اختبار محلي كامل قبل الرفع
  - `git commit` و `git push` بعد التأكد من الشغل

#### 3️⃣ **القاعدة الذهبية للمزامنة 🔒**
- **في نهاية أي جلسة عمل** (سواء سحابة أو محلي):
  - ```bash
    git push -u origin claude/arabic-greeting-DtgO2
    ```
  - **تأكد:** آخر التعديلات وصلت GitHub

- **في بداية أي جلسة جديدة**:
  - ```bash
    git pull origin claude/arabic-greeting-DtgO2
    ```
  - **تأكد:** أجلب آخر التعديلات تجنباً لـ Conflicts

#### 4️⃣ **أوامر مفيدة للمزامنة**
```bash
# معرفة الحالة الحالية
git status

# رؤية الفروقات
git diff

# إضافة جميع التعديلات
git add .

# حفظ التعديلات
git commit -m "رسالة واضحة ومفصلة"

# رفع للـ GitHub
git push -u origin claude/arabic-greeting-DtgO2

# جلب آخر التعديلات
git pull origin claude/arabic-greeting-DtgO2
```

#### 5️⃣ **سيناريوهات عملية**

**السيناريو 1:** أعمل من السحاب، بعدين تنتقل للبيت
```
1. أنا أشتغل (السحاب) → git push
2. أنت تقول "بدأت من الكمبيوتر في البيت"
3. أنت تعمل git pull (تجلب التعديلات)
4. أنت تشتغل محلي → git push عند الانتهاء
```

**السيناريو 2:** أعمل من البيت، بعدين تنتقل برا (آيباد)
```
1. أنت تشتغل محلي → git push عند الانتهاء
2. أنت تقول "بدأت من الآيباد خارج البيت"
3. أنا أشتغل (السحاب) → أسحب آخر التعديلات
4. أنا أشتغل وأضيف → git push فوراً
```

#### 6️⃣ **نقاط أمان وحماية**
- ✅ **بدون تضارب (Conflicts):** لأننا نعمل على فرع واحد مع git pull/push
- ✅ **بدون فقدان أكواد:** كل شيء في GitHub بدون خسارة
- ✅ **مرونة كاملة:** تعمل من أي مكان بأي وقت
- ✅ **تتبع واضح:** git history يشوف كل التعديلات والوقت

---

## ⚠️ قاعدة مهمة:
**قبل رفع أي كود للـ GitHub:**
1. سيتم اختبار الكود محلياً أولاً
2. التأكد من عدم وجود أخطاء
3. التأكد من أن الميزة تعمل بشكل صحيح
4. عرض النتائج عليك للمراجعة
5. فقط بعد موافقتك "OK" يتم الرفع للـ GitHub

## الاتفاقيات الأساسية:

### 1. طريقة العمل

❌ **خطأ:** تعطيني مهمة وأنا أبدأ بسرعة بدون استشارة
✅ **الصحيح:** خطوة بخطوة، استشارة قبل التنفيذ، بدون تسرع أو تخريب
✅ كل خطوة أعرضها عليك قبل ما أكملها

---

### 2. قبل أي تعديل

❌ **خطأ:** أعدل الكود مباشرة بدون backup أو خطة
✅ **الصحيح:** أخذ نسخة قديمة (backup) قبل أي تعديل
✅ عرض الخطة على المستخدم أولاً
✅ انتظار الموافقة "OK" قبل التنفيذ

---

### 3. الرفع للـ GitHub

❌ **خطأ:** رفع الكود بدون موافقة أو بـ force push بدون إذن
✅ **الصحيح:** الرفع بعد الموافقة فقط
✅ مع رسالة commit واضحة ومفصلة
✅ بدون force push بدون إذن صريح

---

### 4. الاختبار

❌ **خطأ:** رفع الكود بدون اختبار محلي
✅ **الصحيح:** التأكد من عدم وجود أخطاء قبل الرفع
✅ شرح النتائج بوضوح للمستخدم

---

### 5. اتفاقية الترابط والإدارة (مهم جداً)

❌ **خطأ:** إضافة ميزة للعميل/الصالون بدون تفعيلها في لوحة الإدارة
✅ **الصحيح:** **القاعدة الذهبية:** أي إضافة مهمة (PIN، بريد، رقم جوال) = يجب تكون في Admin Panel + 2-way sync
✅ الإضافات المهمة أضيفها في التطبيق + الإدارة معاً بدون سؤال
✅ الإضافات البسيطة (ألوان، خطوط) أسأل أولاً قبل التنفيذ
✅ في حالة الشك → اسأل بدل الافتراض

---

### 6. الأمان والبيانات الحساسة

❌ **خطأ:** حذف بيانات العملاء بدون إذن أو التعامل غير آمن مع البيانات
✅ **الصحيح:** بدون حذف بيانات العملاء بدون إذن صريح
✅ حماية البيانات الشخصية والحساسة
✅ التعامل الآمن مع كلمات المرور والـ PIN

---

### 7. الاختبار على جميع الأجهزة

❌ **خطأ:** رفع ميزة بدون اختبارها على جميع الأجهزة (قد تعمل على iPhone بس مو على iPad)
✅ **الصحيح:** اختبار على: iPhone + iPad + Android + الكمبيوتر
✅ التأكد من عمل الميزة على جميع الأجهزة
✅ **بدون رفع** حتى نتأكد من النتائج على الكل

---

### 8. توثيق التغييرات

❌ **خطأ:** رسالة commit غير واضحة مثل "fix" أو "update"
✅ **الصحيح:** رسالة commit واضحة ومفصلة
✅ شرح **السبب** وليس فقط الماذا
✅ ربط التغيير بالمشكلة إن وجدت

---

### 9. سجل النسخ الاحتياطية

❌ **خطأ:** حذف الملفات القديمة بعد التعديل → ما نقدر نرجع إذا حصلت مشكلة
✅ **الصحيح:** احتفظ بنسخة قديمة من أي تعديل مهم
✅ سهولة الرجوع إذا حدثت مشكلة
✅ لا تحذف الملفات القديمة بدون سبب

---

### 10. التوافقية والأداء

❌ **خطأ:** إضافة ميزة تبطي التطبيق أو تكسر توافقيتها مع أجهزة قديمة
✅ **الصحيح:** بدون إبطاء التطبيق
✅ التأكد من التوافقية مع جميع إصدارات التطبيق
✅ اختبار الأداء قبل الرفع

---

### 11. خطوات النشر الموحدة (مهم جداً)

❌ **خطأ:** رفع الكود وتقول "وصل التحديث" بدون ما تعمل merge أو تنتظر Vercel deployment

✅ **الصحيح:** عندما تقول "ارفع"، أتبع هذه الخطوات **بالتسلسل:**

**1. اختبر محلياً** - بدون أخطاء
**2. Git commit** - برسالة واضحة
**3. Git push** - للـ GitHub
**4. ادخل GitHub وشيك الـ PR:**
   - هل في "Merge pull request" button أخضر؟
   - هل يقول "Able to merge"؟
**5. شيك هذه النقاط قبل الـ merge:**
   - ✅ "All checks have passed" (علامات خضراء) - الـ CI checks نجحت
   - ✅ "This branch was successfully deployed" - يظهر في الـ PR
   - ✅ "No conflicts with base branch" - ما في conflicts
**6. اضغط "Merge pull request"** - الزر الأخضر
**7. انتظر رسالة البريد من Vercel** - تأكد من وصول الـ deployment
**8. أخبرك:** "التحديث وصل بنجاح" - **فقط بعد الـ deployment**

✅ **بدون انتظار طويل** - أعمل كل شيء بسرعة
✅ **شيك كل نقطة** - ما أسكيب أي خطوة
✅ **لو في conflict:** أحله من GitHub مباشرة
✅ ما أقول "وصل" إلا لما كل شيء تمام

---

### 12. التواصل في الحالات الطارئة

❌ **خطأ:** انتظار الوقت المحدد رغم وجود مشكلة غير متوقعة (Vercel error، GitHub conflict، bug)

✅ **الصحيح:** لو حصلت مشكلة غير متوقعة → أخبرك **فوراً**
✅ أشرح السبب والحل الفوري
✅ ما أستمر بالشغل قبل الموافقة
✅ أمثلة: Vercel error، GitHub conflict معقد، bug في الكود → أخبرك فوراً

---

### 13. تسهيل سير العمل (للكفاءة)

❌ **خطأ:** تقاطعك المتكرر لي أثناء الشغل، أو قطع السياق قبل انتهاء المهمة

✅ **الصحيح:** 
   - أعطني **السياق الكامل** في البداية - ما تقطعني
   - **ثق بالعملية** - لو بدأت شيء، خليني أكمله
   - **لا تقاطع متكرر** - لو قلت "باشتغل"، انتظر النتيجة
   - **ملاحظات واضحة** - لو في نقاط مهمة، قولها من البداية
   - **صبر على الوقت** - بعض الأشياء تحتاج وقت أكثر

✅ **النتيجة:** شغل أسرع وأفضل، أقل أخطاء، تحديثات توصل بسرعة

---

### 14. التركيز على المطلوب فقط (Focus & Scope)

❌ **خطأ:** تعطيني مهمة واحدة فأشتغل على 5-10 أشياء لم تطلبها (مثل إضافة تحسينات غير مطلوبة) → يفسد المشروع

✅ **الصحيح:** أشتغل على **بالضبط** اللي طلبته بس، بدون إضافات أو تحسينات إضافية
✅ لو رأيت تحسينات أخرى **أسألك أولاً قبل ما أشتغل عليها**
✅ التركيز الكامل على المطلوب فقط

---

### 15. اسأل عند عدم الفهم (Clarification First)

❌ **خطأ:** مثال: تطلب "رمز PIN بسيط يسهل على العميل" وأنا أشتغل على شيء مختلف من تخميني → جلسنا يومين نحل المشكلة

✅ **الصحيح:** 
   - **بدون تخمين** - إذا فهمت شيء غير واضح، ما أشتغل على أساسه
   - **اسأل فوراً** - بمجرد ما ابدأ أشتغل وفيها نقطة غير واضحة
   - **ما أنتظر يومين** - أسأل فوراً لما نفهم بعض تمام
   - **أستمر** لين نوصل لحل يرضيك ويرضيني

✅ **أي نقطة غير واضحة = أسأل فوراً، ما أشتغل على افتراضات**

---

### 16. أنت اللي تسأل (Proactive Questions)

❌ **خطأ:** تنتظر منك السؤال دايماً، وأنا ما أسأل

✅ **الصحيح:**
   - **أنا** أسأل أسئلة استباقية لأفهم احتياجاتك بشكل أفضل
   - "هل تقصد...؟" و "هل بدك...؟" و "شنو رأيك في..."
   - ما أنتظر تشرح كل شيء - أنا اللي أسأل
   - أطلب التوضيح قبل ما ابدأ الشغل

✅ **الصحيح:**
   - **جميع الردود** تكون بالعربية دائماً

### 18. معيار التطبيقات الكبرى

❌ **خطأ:** إضافة ميزة مباشرة بدون مراجعة كيف أضافتها التطبيقات الاحترافية

✅ **الصحيح:** مع كل ميزة جديدة، أشرح لك:
   - **كيف كنا نسويها** — طريقتنا الحالية
   - **كيف تسويها التطبيقات الكبرى** — مثل Airbnb وTalabat وBooking
   - **ما الفرق** — ليش طريقتهم تتجنب أخطاء المستقبل

✅ الهدف: نكشف الأخطاء الكبيرة **الآن** قبل ما تصير — حتى أي خطأ في المستقبل يكون صغير وسهل الحل
✅ أنت تقرر بعد الشرح — نعتمد الطريقة الاحترافية أم لا

### 19. التزامن اللحظي الثلاثي (Real-time Sync)

❌ **خطأ:** إضافة ميزة تعمل من طرف واحد فقط أو تحتاج تحديث يدوي

✅ **الصحيح:** أي ميزة جديدة تمس البيانات لازم تكون **لحظية في ثلاثة اتجاهات:**
- ✅ **عميل ↔ صالون** — الحجوزات، التقييمات، الإلغاء، التعديل
- ✅ **صالون ↔ إدارة** — الحجوزات، الإشعارات، الإعدادات
- ✅ **عميل ↔ إدارة** — الإشعارات، البيانات

✅ **الأدوات المستخدمة:** Supabase Realtime + Polling 10 ثواني + visibilitychange
✅ **أي ميزة جديدة** = تتحقق من الثلاثة اتجاهات قبل الرفع
✅ **في حالة الشك** → اسأل بدل الافتراض

## 🔒 20. القواعس الذهبية لاستعلامات قاعدة البيانات (NON-NEGOTIABLE)

### المقدمة
هذه القواعس **إلزامية 100%** — لا استثناءات. الهدف: منع تسرب البيانات مثل حادثة الـ 18GB Egress.

---

### القاعدة 1️⃣: Zero Select - ممنوع `select('*')`

```javascript
// ❌ ممنوع منعاً باتاً
sb("bookings", "GET", null, "?select=*")
sb("customers", "GET", null, "?select=*")

// ✅ يجب تحديد الأعمدة المطلوبة فقط
sb("bookings", "GET", null, "?select=id,customer_id,salon_id,date,status")
sb("customers", "GET", null, "?select=id,name,phone,email")
```

**السبب:** جلب جميع الأعمدة يستهلك بيانات غير ضرورية ويزيد الـ Egress بشكل هائل.

---

### القاعدة 2️⃣: Mandatory Filter - فلتر محدد إلزامي

أي استعلام لـ `bookings`, `messages`, `waiting_list` **يجب** أن يحتوي على WHERE:

```javascript
// ❌ ممنوع - بدون فلتر (تحميل جميع البيانات!)
sb("bookings", "GET", null, "?select=id,date&limit=100")

// ✅ يجب تحديد salon_id أو customer_id
sb("bookings", "GET", null, `?select=id,date&salon_id=eq.${salonId}&limit=100`)
sb("bookings", "GET", null, `?select=id,date&customer_id=eq.${customerId}&limit=100`)

// ✅ أو customer_phone للعملاء
sb("bookings", "GET", null, `?select=id,date&customer_phone=eq.${phone}&limit=100`)
```

**السبب:** الاستعلامات بدون فلتر تحمل جميع البيانات، مما يسبب تسرب بيانات وزيادة استهلاك النطاق الترددي.

---

### القاعدة 3️⃣: Pagination/Limit - استخدام Limit دائماً

إذا كانت البيانات قد تكون كثيرة، استخدم `limit`:

```javascript
// ❌ بدون limit - قد يحمل ملايين الصفوف
sb("messages", "GET", null, `?select=id,text&chat_id=eq.${chatId}`)
sb("reviews", "GET", null, `?select=*&salon_id=eq.${salonId}`)

// ✅ مع limit وorder
sb("messages", "GET", null, `?select=id,text,sender_id,created_at&chat_id=eq.${chatId}&order=created_at.desc&limit=50`)
sb("reviews", "GET", null, `?select=id,rating,text,created_at&salon_id=eq.${salonId}&order=created_at.desc&limit=20`)
```

**الـ Limits الموصى بها:**
- Messages/Chat: `limit=50`
- Reviews: `limit=20`
- Bookings: `limit=100-1000`
- Customers: `limit=500`

**السبب:** تحديد الكمية يمنع تحميل البيانات غير الضرورية.

---

### القاعدة 4️⃣: RLS Awareness - افترض الحماية مفعلة دائماً

افترض أن **Row Level Security** مفعلة على جميع الجداول:

```javascript
// ✅ أضف session_token في Headers
const headers = {
  'Authorization': `Bearer ${sessionToken}`,
};

// ✅ تأكد أن الاستعلام يحترم صلاحيات المستخدم
// (الـ RLS سيرفضه تلقائياً إذا لم يكن للمستخدم صلاحية)
sb("bookings", "GET", headers, `?select=id,date&salon_id=eq.${userSalonId}`)

// ❌ لا تحاول الالتفاف على RLS بـ workarounds
// لا تستخدم service_role في الـ frontend
// لا تطلب بيانات المستخدمين الآخرين
```

**السبب:** RLS هو خط الدفاع الأخير ضد تسرب البيانات.

## 📜 21. ميثاق الأداء والوقاية التقنية (Performance & Technical Prevention Protocol)

### البيان الملزم:

يا كلود، بصفتك المهندس المسؤول عن هذا المشروع، نلتزم في كل تعديل أو إضافة (سواء كود جديد أو أعمدة قاعدة بيانات Supabase) بالقواعد الذهبية التالية **التي لا تهاون فيها:**

---

### القاعدة 1️⃣: الاستعلام الجراحي (The Surgical Fetch Rule)

```javascript
// ❌ ممنوع منعاً باتاً
sb.from('bookings').select('*')
sb.from('customers').select('*')

// ✅ يجب تحديد الأعمدة فقط
sb.from('bookings').select('id, customer_id, salon_id, date, status, created_at')
sb.from('customers').select('id, name, phone, email, avatar_url')
```

**الالتزامات الإلزامية:**

1. **يمنع منعاً باتاً استخدام `select=*`** — تحديد الأعمدة المطلوبة فقط دائماً
2. **كل استعلام يجب أن يحتوي على `limit` منطقي** (50، 100، 500) لمنع سحب كامل الجدول
3. **الحد الأدنى للـ limits:**
   - Messages/Chat: `limit=50`
   - Reviews/Ratings: `limit=20`
   - Bookings: `limit=100-500`
   - Customers: `limit=500`
   - Salons: `limit=500`

**السبب:** منع تسرب البيانات والـ Egress مثل حادثة الـ 18GB السابقة.

### القاعدة 2️⃣: مكافحة التكرار (The Anti-Duplication Rule)

```javascript
// ❌ ممنوع - نفس الـ API في مكانين
useEffect(() => {
  loadAppSettings() // هنا
}, [])

useEffect(() => {
  loadAppSettings() // وهنا أيضاً!
}, [])

// ✅ الصحيح - مكان واحد مركزي
useEffect(() => {
  loadAppSettings() // مكان واحد فقط
}, [])
```

**الالتزامات الإلزامية:**

1. **يمنع استدعاء نفس الـ API أو Function في أكثر من useEffect**
2. **أي loadData يجب أن يتم في مكان واحد مركزي** (Single Source of Truth)
3. **توزيع البيانات عبر Context API أو Props** — بدون تكرار الطلبات
4. **تتبع الحالة:** استخدم state centralized لتجنب استدعاءات متعددة

**السبب:** منع استهلاك النطاق الترددي المجاني بـ API calls متكررة.

### القاعدة 3️⃣: نظافة الخلفية (The Clean Background Rule)

```javascript
// ❌ ممنوع - بدون cleanup
useEffect(() => {
  const channel = supabase
    .channel('bookings')
    .on('*', payload => handleChange(payload))
    .subscribe()
}, [])

// ✅ يجب cleanup فوراً
useEffect(() => {
  const channel = supabase
    .channel('bookings')
    .on('*', payload => handleChange(payload))
    .subscribe()
  
  return () => channel.unsubscribe() // cleanup!
}, [])
```

**الالتزامات الإلزامية:**

1. **أي Realtime Channel يتم فتحه يجب أن يُغلق باستخدام `unsubscribe()`** في الـ cleanup function
2. **يمنع استدعاء أي background task** (مثل FCM notifications) بشكل متكرر عند فشل الاتصال
3. **تفعيل Circuit Breaker البسيط:**
   - عد محاولات الفشل
   - توقف بعد 3 محاولات فشل
   - انتظر 30 ثانية قبل إعادة المحاولة

```javascript
let failureCount = 0
const MAX_FAILURES = 3

const sendNotification = async () => {
  try {
    await fcm.send(...)
    failureCount = 0 // reset
  } catch (error) {
    failureCount++
    if (failureCount >= MAX_FAILURES) {
      console.warn('Circuit Breaker: Stopping retries after 3 failures')
      return // توقف!
    }
  }
}
```

**السبب:** منع memory leaks والاستهلاك المستمر للموارد من الاتصالات المتروكة.

```javascript
// ❌ ممنوع - تحميل الصور الخام (Original Size)
const imageUrl = salon.image_url // الصورة الكاملة (عالية الدقة)

// ✅ استخدم الصور المحسّنة (Thumbnails)

**الالتزامات الإلزامية:**

1. **يمنع تحميل الصور الخام (Original Size)** — استخدم روابط محسّنة دائماً (Thumbnails/Optimized)

## 🧹 22. منع تكرار الأكواد ونظافة الملف (Zero Code Duplication)

### المبدأ الأساسي:
**الكود يعيش في مكان واحد فقط — لا استثناءات.**

---

### القاعدة 1️⃣: النقل الحقيقي (Move ≠ Copy)

عندما يُنقل كود من مكان إلى آخر، يُحذف من مكانه القديم فوراً:

```
// ❌ ممنوع — الكود موجود في مكانين
[المكان القديم] ← الكود لا يزال هنا
[المكان الجديد] ← ونسخة منه هنا أيضاً

// ✅ الصحيح — الكود في مكان واحد فقط
[المكان القديم] ← محذوف
[المكان الجديد] ← هنا فقط
```

**القاعدة:** النقل = حذف من الأصل + إضافة في الهدف. ليس نسخاً.

---

### القاعدة 2️⃣: حذف الميزة = حذف كودها كاملاً

عندما تُحذف ميزة أو عنصر من التطبيق، يُحذف كودها كاملاً من الملف:

```
// ❌ ممنوع — ميزة محذوفة لكن كودها بقي
{/* تم حذف هذا الزر */}
// <button onClick={...}>زر قديم</button>

// ✅ الصحيح — الكود محذوف بالكامل، لا أثر له
[لا شيء — الكود اختفى تماماً]
```

**السبب:** الكود المعلّق (commented out) والكود الميت (dead code) يضخّمان الملف ويسببان تضارباً في التحديثات القادمة.

---

### القاعدة 3️⃣: تصميم موحّد = كود مشترك

إذا كان عنصران بنفس التصميم والوظيفة، فكودهما يكون مشتركاً — ليس مكرراً:

```
// ❌ ممنوع — نفس النافذة معرّفة مرتين في مكانين
// نافذة حذف الحساب في Drawer (150 سطر)
// نافذة حذف الحساب في custDash (150 سطر مطابقة)

// ✅ الصحيح — مكوّن واحد مشترك أو تصميم موحّد
// يُستدعى من كلا المكانين
```

---

### ✅ قائمة تحقق إلزامية قبل كل Commit:

- [ ] هل نقلتُ كوداً؟ → تأكد أنه حُذف من مكانه القديم
- [ ] هل حذفتُ ميزة؟ → تأكد أن كودها اختفى كاملاً من الملف
- [ ] هل في عنصرين بنفس التصميم؟ → يجب توحيدهما أو مشاركة كودهما
- [ ] هل حجم الملف يكبر بلا سبب؟ → راجع الكود المكرر أو الميت
