# DORK PROJECT CHARTER — الميثاق الشامل للمشروع

> **نسخة:** 2.0 — مراجعة موسّعة  
> **تاريخ:** 2026-06-29  
> **المشروع:** Dork Barbershop App — تطبيق حجوزات صالونات الحلاقة  
> **البنية التقنية:** React/Vite (PWA) + Supabase + Vercel API + Next.js Admin

---

## نبذة عن المشروع

Dork هو تطبيق حجوزات صالونات حلاقة يتكون من ثلاثة أطراف:
- **تطبيق العميل** — React/Vite PWA (بدون TypeScript، عربي RTL)
- **تطبيق الصالون** — مدمج في نفس الـ App.jsx
- **لوحة الإدارة** — Next.js 14 + TypeScript + Tailwind (App Router)

قاعدة البيانات Supabase مشتركة بين الأطراف الثلاثة. النشر على Vercel.

## القسم الثاني: بروتوكولات الاختبار والتحكم (34 نقطة)

### أ- اختبارات الأداء (Performance)
42. ضمان سرعة الاستجابة — **تفصيل:** TTFB < 200ms، FCP < 1.5s، LCP < 2.5s (وليس فقط "< 500ms" عام).
43. تنفيذ الـ Lazy Loading للمكونات الثقيلة (> 50kb) فقط، مع قياس أثرها على Re-renders.
44. ضغط تلقائي للصور (WebP format) + Responsive Images (srcset) حسب حجم الشاشة.
45. استخدام Caching لبيانات الصالونات الثابتة (localStorage + 24 ساعة TTL).
46. مراقبة استهلاك موارد الـ Edge Functions (Vercel Function Duration).
47. اختبار التحمل (Stress Test) على بيئة **Staging** — معيار النجاح: p95 latency < 800ms، error rate < 0.1% عند 1000 مستخدم.
48. التخلص من أي Re-render غير ضروري (React DevTools Profiler قبل كل إصدار).

### ب- دقة البيانات والتقارير (Data Integrity)
49. مطابقة تامة بين تقارير الأرباح في الأدمن والواقع (تحقق يدوي شهري).
50. مزامنة فورية بين تغييرات الصالون وشاشة العميل (Supabase Realtime).
51. نظام أرشفة تلقائي للحجوزات المنتهية (> 6 أشهر) — يُحدد: هل تُحذف أم تُنقل لجدول archive؟
52. معالجة دقيقة للمناطق الزمنية (Timezone synchronization) — تخزين UTC، عرض بتوقيت المستخدم.
53. تنبيه فوري للأدمن عند فشل أي عملية دفع أو حجز.
54. نظام Audit Logging لكل عملية تعديل أسعار أو حذف بيانات.

### ج- الأمان والاتصال (Security & Connectivity)
55. حماية كاملة لقنوات الاتصال اللحظي (Websockets/Realtime).
56. إغلاق الاتصالات غير النشطة فوراً (cleanup على unmount).
57. صد هجمات الـ Brute Force: حد أقصى 5 محاولات/دقيقة لكل IP على endpoints الدخول.
58. تشفير الـ PIN كود (Hashed + Salted باستخدام PBKDF2 أو bcrypt).
59. سجل نشاط (Action Logs) لكل تحرك حساس (دخول، حجز، تعديل، حذف).

### د- تجربة المستخدم (UI/UX)

## القسم الثالث: نقاط مقترحة إضافية — مستوى "التطبيقات المليونية" (8 نقاط جديدة)

### 76. منع الحجز المزدوج (Idempotency)
**المشكلة:** عند إعادة المحاولة أو ضغط الزر مرتين، يُسجَّل الحجز مرتين.  
**الحل:** UNIQUE constraint في قاعدة البيانات + التحقق قبل كل INSERT.
```sql
ALTER TABLE bookings 
ADD CONSTRAINT unique_active_booking 
UNIQUE (salon_id, customer_phone, date, time_slot)
WHERE status NOT IN ('cancelled', 'rejected');
```
**الأثر:** يمنع تكرار البيانات ويحمي ثقة المستخدم في الحجوزات.

---

### 77. رصد الأعطال الصامتة (Crash Reporting)
**المشكلة:** عندما يواجه مستخدم خطأً في الإنتاج، لا تعرف به إلا من الشكاوى.  
**الحل:** دمج Sentry أو Highlight.io (مجاناً حتى حد معين) لالتقاط كل Error مع:
- Stack trace كامل
- معلومات المستخدم (بدون بيانات شخصية)
- تنبيه فوري عند تجاوز 10 أخطاء/دقيقة

**الأثر:** تكتشف الأعطال قبل أن يشتكي المستخدمون.

---

### 78. التدهور الرشيق عند انقطاع الخدمة (Graceful Degradation)
**المشكلة:** إذا توقف Supabase أو Vercel، يرى المستخدم صفحة بيضاء أو خطأ غير مفهوم.  
**الحل:**
- شاشة "الخدمة غير متاحة مؤقتاً" واضحة وجميلة
- الاحتفاظ بالبيانات المحلية (localStorage) لحين عودة الاتصال
- Circuit Breaker يوقف المحاولات بعد 3 فشل متتالي وينتظر 30 ثانية

**الأثر:** تجربة مستخدم محترمة حتى في أسوأ الظروف.

---

### 79. فحص أمان المكتبات تلقائياً (Dependency Security Scanning)
**المشكلة:** مكتبة npm واحدة فيها ثغرة = ثغرة في كامل التطبيق.  
**الحل:** تشغيل `npm audit --audit-level=high` في كل CI/CD pipeline تلقائياً — يرفض البناء عند وجود ثغرات High/Critical.  
**الأثر:** تحمي المستخدمين من ثغرات المكتبات الخارجية.

---

### 80. حماية مزدوجة لنقاط الـ API (CORS + CSP Headers)
**المشكلة:** أي موقع خارجي يمكنه نظرياً استدعاء الـ API.  
**الحل:**
```javascript
// في Vercel API
res.setHeader('Access-Control-Allow-Origin', 'https://dork-app.vercel.app');
res.setHeader('Content-Security-Policy', "default-src 'self'; script-src 'self' 'unsafe-inline' https://www.gstatic.com https://apis.google.com");
```
**الأثر:** يمنع أي طرف ثالث من استدعاء API التطبيق.

---

### 81. فهارس قاعدة البيانات على الأعمدة الأكثر استعلاماً (Database Indexes)
**المشكلة:** مع نمو البيانات (آلاف الحجوزات)، الاستعلامات بدون indexes تصبح بطيئة جداً.  
**الحل:**
```sql
CREATE INDEX idx_bookings_salon_date ON bookings(salon_id, date);
CREATE INDEX idx_bookings_customer_phone ON bookings(customer_phone);
CREATE INDEX idx_bookings_status ON bookings(status) WHERE status = 'pending';
CREATE INDEX idx_salons_phone ON salons(phone);
CREATE INDEX idx_salons_status ON salons(status) WHERE status = 'approved';
```
**الأثر:** يمنع التباطؤ مع نمو قاعدة البيانات — الفرق بين 10ms و2000ms.

### 82. خطة الطوارئ التشغيلية (Incident Response Runbook)
**المشكلة:** لا يوجد بروتوكول واضح لماذا يحدث عندما يتوقف التطبيق.  
**المطلوب:** مستند واحد يجيب على:
- كيف أعرف أن التطبيق توقف؟ (Monitoring/Alert)
- ما أول 3 خطوات للتشخيص؟
- من أتصل به (Supabase Support? Vercel Support?)
- كيف أُعيد التشغيل في أسرع وقت؟
- متى أُبلّغ المستخدمين؟

**الأثر:** يقلل وقت التوقف من ساعات إلى دقائق.

---

### 83. Core Web Vitals كـ KPIs إلزامية (لا مجرد توصيات)
**المشكلة:** النقطة 42 الأصلية عامة جداً ("< 500ms").  
**الحل:** معايير محددة وقابلة للقياس في كل إصدار:

| المقياس | الهدف | الحد الأقصى المقبول | أداة القياس |
|---------|-------|---------------------|-------------|
| LCP (Largest Contentful Paint) | < 1.5s | < 2.5s | Lighthouse |
| CLS (Cumulative Layout Shift) | < 0.05 | < 0.1 | Lighthouse |
| INP (Interaction to Next Paint) | < 100ms | < 200ms | Chrome DevTools |
| TTFB (Time to First Byte) | < 150ms | < 200ms | Vercel Analytics |

**القاعدة:** إذا تجاوز أي مقياس الحد الأقصى → لا رفع للإنتاج حتى يُحل.  
**الأثر:** يضمن أن التطبيق يبقى سريعاً مع كل إصدار جديد.

## أسئلة للنقاش (مع Gemini أو أي مستشار تقني)

1. **الحجز المزدوج (النقطة 76):** هل UNIQUE constraint وحده كافٍ أم نحتاج Idempotency Key في الـ API أيضاً؟
2. **Crash Reporting (النقطة 77):** Sentry مقابل Highlight.io — أيهما أنسب لـ React/Vite PWA مع Supabase؟
3. **Staging (النقطة 28):** ما التكلفة الفعلية لـ Supabase project ثانٍ للـ Staging؟ هل هناك بديل أرخص؟
4. **Database Indexes (النقطة 81):** هل Supabase يدير الـ indexes تلقائياً لـ Primary Keys فقط؟ وهل يحتاج إذن إضافي لإضافة indexes يدوياً؟
5. **RLS (النقطة 39 + 72):** في تطبيق حجوزات متعدد الأطراف (عميل/صالون/أدمن)، ما الأنسب: RLS rules معقدة أم استخدام Service Role مع Vercel API فقط؟
