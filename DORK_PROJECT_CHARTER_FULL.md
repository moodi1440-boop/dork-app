Dork هو تطبيق حجوزات صالونات حلاقة يتكون من ثلاثة أطراف:
- **تطبيق العميل** — React/Vite PWA (بدون TypeScript، عربي RTL)
- **تطبيق الصالون** — مدمج في نفس الـ App.jsx
- **لوحة الإدارة** — Next.js 14 + TypeScript + Tailwind (App Router)

قاعدة البيانات Supabase مشتركة بين الأطراف الثلاثة. النشر على Vercel.

## القسم الثالث: نقاط مقترحة إضافية — مستوى "التطبيقات المليونية" (8 نقاط جديدة)

### 76. منع الحجز المزدوج (Idempotency)
**المشكلة:** عند إعادة المحاولة أو ضغط الزر مرتين، يُسجَّل الحجز مرتين.  
**الحل (مرحلتان — معتمد من مراجعة Gemini):**

**أ. UNIQUE constraint في قاعدة البيانات** — خط الدفاع الأخير، إلزامي:
```sql
ALTER TABLE bookings 
ADD CONSTRAINT unique_active_booking 
UNIQUE (salon_id, customer_phone, date, time_slot)
WHERE status NOT IN ('cancelled', 'rejected');
```

**ب. Idempotency Key على مستوى الـ API** — ضروري في الأنظمة الموزعة:
> السبب: العميل قد يضغط زر "تأكيد الحجز"، يُرسل الطلب، لكن الإنترنت ينقطع قبل وصول الرد. التطبيق يُعيد المحاولة تلقائياً أو يدوياً من العميل — بدون مفتاح idempotency، يُسجَّل الحجز مرتين قبل أن يصل الطلب الثاني لقاعدة البيانات حتى.

```javascript
// الكلاينت يُولّد مفتاحاً فريداً لكل محاولة حجز
const idempotencyKey = crypto.randomUUID();
await fetch('/api/create-booking', {
  method: 'POST',
  headers: { 'Idempotency-Key': idempotencyKey },
  body: JSON.stringify(bookingData)
});

// الـ API يتحقق من المفتاح قبل أي INSERT
const existing = await sb('idempotency_keys','GET',null,`?key=eq.${idempotencyKey}&limit=1`);
if (existing.length) return res.status(200).json(existing[0].response); // نفس النتيجة القديمة
```

**الأثر:** UNIQUE constraint يمنع البيانات الفاسدة وصول لقاعدة البيانات، وIdempotency Key يمنع حتى محاولة الإرسال المكررة من الأساس — حماية مزدوجة.

### 77. رصد الأعطال الصامتة (Crash Reporting)
**المشكلة:** عندما يواجه مستخدم خطأً في الإنتاج، لا تعرف به إلا من الشكاوى.  
**الحل المعتمد: Highlight.io** (وليس Sentry) — توصية Gemini:
> السبب: Highlight.io يجمع بين تتبع الأخطاء (Error Tracking) و**Session Replay** — رؤية "ماذا فعل المستخدم بالضبط قبل أن ينهار التطبيق" بالضبط (كل ضغطة وكل شاشة). في تطبيق حجوزات، هذا يعني معرفة هل المستخدم كان في خطوة اختيار الموعد أم الدفع عندما حصل الخطأ — لا يُقدَّر بثمن لتشخيص أسرع.

**يُسجَّل:**
- Stack trace كامل
- Session Replay (تسلسل تفاعل المستخدم قبل الخطأ)
- معلومات المستخدم (بدون بيانات شخصية حساسة)
- تنبيه فوري عند تجاوز 10 أخطاء/دقيقة

**الأثر:** تكتشف الأعطال قبل أن يشتكي المستخدمون، وتُشخّصها بسرعة أكبر بفضل Session Replay.

### 78. التدهور الرشيق عند انقطاع الخدمة (Graceful Degradation)
**المشكلة:** إذا توقف Supabase أو Vercel، يرى المستخدم صفحة بيضاء أو خطأ غير مفهوم.  
**الحل:**

- Circuit Breaker يوقف المحاولات بعد 3 فشل متتالي وينتظر 30 ثانية

**الأثر:** تجربة مستخدم محترمة حتى في أسوأ الظروف.

### 81. فهارس قاعدة البيانات على الأعمدة الأكثر استعلاماً (Database Indexes)
**المشكلة:** مع نمو البيانات (آلاف الحجوزات)، الاستعلامات بدون indexes تصبح بطيئة جداً.  
**تأكيد من Gemini:** Supabase/PostgreSQL ينشئ Indexes تلقائياً **للـ Primary Keys فقط** — أي عمود آخر (salon_id، customer_phone، status...) يحتاج إنشاء index يدوي. بدون هذا، الفرق في الأداء قد يصل لعشرات الأضعاف مع نمو البيانات.  
**الحل:**
```sql
CREATE INDEX idx_bookings_salon_date ON bookings(salon_id, date);
CREATE INDEX idx_bookings_customer_phone ON bookings(customer_phone);
CREATE INDEX idx_bookings_status ON bookings(status) WHERE status = 'pending';
CREATE INDEX idx_salons_phone ON salons(phone);
CREATE INDEX idx_salons_status ON salons(status) WHERE status = 'approved';
```
**الأثر:** يمنع التباطؤ مع نمو قاعدة البيانات — الفرق بين 10ms و2000ms.

### 83. Core Web Vitals كـ KPIs إلزامية (لا مجرد توصيات)
**المشكلة:** النقطة 42 الأصلية عامة جداً ("< 500ms").  
**الحل:** معايير محددة وقابلة للقياس في كل إصدار:

| المقياس | الهدف | الحد الأقصى المقبول | أداة القياس |
|---------|-------|---------------------|-------------|
| LCP (Largest Contentful Paint) | < 1.5s | < 2.5s | Lighthouse |
| CLS (Cumulative Layout Shift) | < 0.05 | < 0.1 | Lighthouse |
| INP (Interaction to Next Paint) | < 100ms | < 200ms | Chrome DevTools |
| TTFB (Time to First Byte) | < 150ms | < 200ms | Vercel Analytics |

**القاعدة:** إذا تجاوز أي مقياس الحد الأقصى → لا رفع للإنتاج حتى يُحل.  
**الأثر:** يضمن أن التطبيق يبقى سريعاً مع كل إصدار جديد.

| 3 | بيئة Staging | **Supabase Free Tier — مشروع ثانٍ بتكلفة صفر** (مشروع Production + مشروع Staging) |
| 4 | Database Indexes | Supabase ينشئ index تلقائياً **للـ Primary Key فقط** — كل عمود آخر (salon_id، customer_phone، status) يحتاج index يدوي إلزامي |
| 5 | RLS مقابل Service Role | **RLS فقط في الـ Client — حظر قطعي لـ Service Role في الـ Client.** Service Role يبقى محصوراً بـ Vercel API (server-side) كما هو الحال حالياً |
