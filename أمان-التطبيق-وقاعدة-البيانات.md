# أمان التطبيق وقاعدة البيانات

> **الفرق عن `تقرير-أمان-RLS.md`:** ذاك الملف سجل تاريخي (قصة كل حادثة أمنية وإصلاحها بترتيب زمني). هذا الملف **الوضع الحالي + قائمة حية**.
>
> **يُسجَّل هنا دائماً:**
> 1. **كل شي يرفع نسبة الأمان** (بالكود أو بقاعدة البيانات) — قسم "طبقات الحماية المطبَّقة"
> 2. **كل نقطة/طريقة يحاول منها المخترقون الدخول** (IDOR، XSS، تجاوز Rate Limiting، سرقة الجلسة، إساءة استخدام منطق العمل، تسريب أسرار، وغيرها) — قسم "دليل الفحص الذاتي" + جدول حالة التنفيذ لكل واحدة (نُفِّذت/اختُبرت أو لسا)
>
> يُحدَّث باستمرار — أي نقطة جديدة بأي من النوعين (حماية جديدة، أو طريقة اختراق جديدة نكتشفها أو نفكر فيها) تُضاف فوراً، سواء نُفِّذت مباشرة أو انتظرت التنفيذ لاحقاً.
>
> آخر تحديث: 2026-07-14

## ✅ طبقات الحماية المطبَّقة فعلياً

### بالكود
- تشفير (hash) رمز دخول الصالون بالسيرفر — `api/_lib/owner-session.js` (`hashOwnerPin`)
- تحقق حقيقي بالسيرفر لرمز العميل (خطوة أولى) — `api/customer-auth.js`
- Rate limiting على محاولات تسجيل الدخول — `api/_lib/rate-limit.js`، مُفعَّل بـ`owner-auth.js`, `register-salon.js`
- قفل مؤقت بعد محاولات فاشلة متكررة (5 محاولات → قفل 10 دقائق) + رسالة توضح الوقت المتبقي
- حماية من الضغط المزدوج بنماذج التسجيل (بعد إصلاح 2026-07-14)
- فصل صلاحيات الحذف الذاتي (قابل للاسترجاع) عن الحذف النهائي بلوحة الإدارة

### بقاعدة البيانات (سيبابيس)
- RLS مفعّل على كل الجداول + "Enable automatic RLS" على DORK-TEST (يمنع نسيان تفعيله بجدول جديد)
- GRANT مطابق تماماً لكل RLS policy (بعد تدقيق شامل 2026-07-07)
- `ALTER DEFAULT PRIVILEGES` دائم لصلاحيات الـsequences — يمنع تكرار فخّ "sequence بدون GRANT" لأي جدول مستقبلي
- إخفاء أعمدة حساسة (كلمة مرور الصالون، owner_pin_hash) عن anon/authenticated عبر GRANT على مستوى العمود (column-level)
- DORK وDORK-TEST متطابقان 100% هيكلةً وصلاحياتٍ (يسمح باختبار آمن قبل أي تغيير خطير على الإنتاج)
- إخفاء `pin_hash`/`pin_fails`/`pin_locked_until` لجدول `customers` عن anon/authenticated (2026-07-14) — يمنع قراءة/تعديل الرمز السري المُشفّر مباشرة عبر API خارج التطبيق

## 🔍 دليل الفحص الذاتي (لأحمد — بأدوات المتصفح، بدون برامج) + جميع نقاط الاختراق المحتملة

النص الكامل بالخطوات موجود بـ`تقرير-أمان-RLS.md` (قسم 2026-07-14). ملخص الاختبارات:

1. **XSS** — إدخال `<img src=x onerror=alert('XSS')>` بحقل نصي يظهر لطرف ثاني، وتأكيد إنه ما يُنفَّذ
2. **IDOR** — تعديل رقم ID بطلب شبكي (Network tab) والتأكد ما يرجّع بيانات حساب ثاني
3. **تجاوز Rate Limiting** — محاولات دخول خاطئة متكررة، تأكيد القفل يشتغل
4. **أمان الكوكيز + CSRF** — **لصاحب الصالون:** F12 → Application → Cookies → `dork_owner_session`، تأكيد HttpOnly + Secure + SameSite (✅ تحقّق كودي 2026-07-14 أكّد الثلاثة صحيحة). **للعميل:** ⚠️ ما فيه كوكي أصلاً — الجلسة بـlocalStorage (مفتاح `sb-...-auth-token`) عبر مكتبة Supabase، بدون حماية HttpOnly — ضعف معماري معروف مسبقاً، يخلي اختبار XSS (بند 1) حرج بشكل خاص لجهة العميل
5. **تسريب الأسرار** — بحث بكود المصدر عن `service_role`, `SECRET`, `PRIVATE_KEY`
6. **إساءة استخدام منطق العمل** — تجاوز حد استخدام كود خصم، أو حجز متعارض
7. **ثغرات مكتبات جاهزة قديمة (`npm audit`)** — ✅ **نُفِّذ فعلياً 2026-07-14** (مو اختبار متصفح، شغّله كلود مباشرة): `dork-app` صفر ثغرات، `admin/` 14 ثغرة (تفاصيل بجدول "نقاط أمان عاجلة" أعلاه وبـ`تقرير-أمان-RLS.md`)

⚠️ اختبارات الكتابة (XSS، استغلال الأكواد) على صالون تجريبي فقط، مو "الوادي" الحقيقي.

**✅ قرار أحمد (2026-07-14):** تُنفَّذ اختبارات المتصفح (1-6) بجلسة لاحقة (بعدين) — يبدأ بتسجيل صالون تجريبي مخصّص للاختبار (زي "لقمان" سابقاً)، ثم يجرَّب كل اختبار بالترتيب واحد واحد بالتنسيق المباشر مع كلود، ويُسجَّل نتيجة كل واحد هنا فور تنفيذه.

**حالة التنفيذ (تُحدَّث أول بأول):**

| # | الاختبار | الحالة |
|---|---|---|
| 1 | XSS | ⏳ لم يُنفَّذ بعد |
| 2 | IDOR | ⏳ لم يُنفَّذ بعد |
| 3 | تجاوز Rate Limiting | ⏳ لم يُنفَّذ بعد (ملاحظة: قفل الدخول بعد 5 محاولات تأكد شغّال فعلاً بنقطة #19 — لكن هذا اختبار محاولة تجاوزه فعلياً، مو نفس الشي) |
| 4 | أمان الكوكيز + CSRF (HttpOnly/Secure/SameSite) | 🟡 **جزئي 2026-07-14** — تحقّق كودي (مو اختبار متصفح حي بعد): صاحب الصالون ✅ محمي بالكامل (HttpOnly+Secure+SameSite). العميل ❌ بدون حماية HttpOnly (localStorage) — ضعف معماري معروف |
| 5 | تسريب الأسرار بكود المتصفح | 🟡 **جزئي 2026-07-14** — تحقّق كودي أوّلي: صفر نتائج لـ`service_role` بكود الواجهة (App.jsx وصفحات admin)، ومفتاح service_role بلوحة الإدارة مقروء فقط من كود سيرفر (بدون `NEXT_PUBLIC_`) ولا يستورده أي مكوّن client — لسه ناقص فحص حي كامل بالمتصفح المنشور فعلياً |
| 6 | إساءة استخدام منطق العمل (كود خصم/تعارض حجز) | ⏳ لم يُنفَّذ بعد |
| 7 | ثغرات مكتبات جاهزة (`npm audit`) | ✅ **نُفِّذ 2026-07-14** — `dork-app`: صفر؛ `admin/`: 14 ثغرة (إصلاح Next.js متاح وآمن، إصلاح firebase يحتاج قفزة إصدار رئيسي) |

**تدقيق كودي إضافي 2026-07-14 (بطلب أحمد "لازم نطبق النقاط هذي كلها" — مراجعة الكود الفعلي مقابل قائمة الكلمات/الرموز اللي ترفع الأمان):**

| الممارسة | مطبَّقة فعلياً؟ |
|---|---|
| تشفير الرموز السرية (hash) | ✅ نعم — `hashOwnerPin`, `hashCustomerPin` |
| React auto-escape ضد XSS | ✅ نعم — استخدام وحيد لـ`dangerouslySetInnerHTML` بـ`admin/src/app/layout.tsx` وهو نص ثابت (theme init)، مو محتوى مستخدم |
| HttpOnly + Secure + SameSite على كوكي الجلسة | ✅ لصاحب الصالون فقط — ❌ ما يوجد أصلاً لجهة العميل (localStorage بدل كوكي) |
| أسرار السيرفر (`service_role`) غير مسرَّبة للمتصفح | ✅ نعم — صفر استخدام بكود العميل/admin المُقدَّم للمتصفح، ومفتاح الإدارة `SUPABASE_SERVICE_ROLE_KEY` (بدون `NEXT_PUBLIC_`) لا يُستورَد من أي client component |
| `auth.uid()` بسياسات RLS بدل `true` | ❌ لا — هذا بالضبط فجوة Tier 2 المفتوحة |
| GRANT محدّد بأعمدة (column-level) للحساس | ✅ نعم — مطبَّق على `salons` و`customers` |

**الخلاصة:** أغلب "الكلمات/الرموز" المذكورة مطبَّقة فعلاً بالكود، ما عدا فجوتين معروفتين مسبقاً وما تغيّرتا: جلسة العميل (localStorage) وTier 2 (`auth.uid()`).
