## ✅ طبقات الحماية المطبَّقة فعلياً

### بالكود
- تشفير (hash) رمز دخول الصالون بالسيرفر — `api/_lib/owner-session.js` (`hashOwnerPin`)
- تحقق حقيقي بالسيرفر لرمز العميل (خطوة أولى) — `api/customer-auth.js`
- Rate limiting على محاولات تسجيل الدخول — `api/_lib/rate-limit.js`، مُفعَّل بـ`owner-auth.js`, `register-salon.js`
- قفل مؤقت بعد محاولات فاشلة متكررة (5 محاولات → قفل 10 دقائق) + رسالة توضح الوقت المتبقي
- حماية من الضغط المزدوج بنماذج التسجيل (بعد إصلاح 2026-07-14)
- فصل صلاحيات الحذف الذاتي (قابل للاسترجاع) عن الحذف النهائي بلوحة الإدارة

### بقاعدة البيانات (سيبابيس)
- RLS مفعّل على كل الجداول + "Enable automatic RLS" على DORK-TEST (يمنع نسيان تفعيله بجدول جديد)
- GRANT مطابق تماماً لكل RLS policy (بعد تدقيق شامل 2026-07-07)
- `ALTER DEFAULT PRIVILEGES` دائم لصلاحيات الـsequences — يمنع تكرار فخّ "sequence بدون GRANT" لأي جدول مستقبلي
- إخفاء أعمدة حساسة (كلمة مرور الصالون، owner_pin_hash) عن anon/authenticated عبر GRANT على مستوى العمود (column-level)
- DORK وDORK-TEST متطابقان 100% هيكلةً وصلاحياتٍ (يسمح باختبار آمن قبل أي تغيير خطير على الإنتاج)

## ⏳ نقاط أمان معلّقة (لسه ناقصة)

| النقطة | السبب/الشرط |
|---|---|
| **مراجعة أمنية مستقلة (Pentest)** من مختص خارجي | إلزامي قبل أي إطلاق تجاري حقيقي — تفاصيل بـ`تقرير-أمان-RLS.md` (قسم 2026-07-14) |
| ضعف معماري بتسجيل دخول **العميل** | التحقق أساساً محلي (localStorage)، السيرفر يتحقق بالخلفية بصمت بدون ما يمنع الدخول لو فشل — يحتاج تصميم أقوى مستقبلاً |
| فحص IDOR فعلي (هل يقدر مستخدم يشوف بيانات حساب ثاني بتغيير ID؟) | لم يُختبر حياً بعد — دليل الفحص الذاتي أدناه |
| فحص XSS فعلي بكل الحقول النصية المعروضة لطرف ثاني | لم يُختبر حياً بعد |
| فحص أمان الكوكيز (HttpOnly/Secure/SameSite) | لم يُتحقّق منه بعد |
| فحص تسريب أسرار بكود المتصفح المُجمَّع (bundle) | لم يُفحص بعد |
| ثغرات مكتبات جاهزة قديمة (npm audit / Snyk) | لم يُفحص بعد |

## 🔍 دليل الفحص الذاتي (لأحمد — بأدوات المتصفح، بدون برامج)

النص الكامل بالخطوات موجود بـ`تقرير-أمان-RLS.md` (قسم 2026-07-14). ملخص الاختبارات الستة:

1. **XSS** — إدخال `<img src=x onerror=alert('XSS')>` بحقل نصي يظهر لطرف ثاني، وتأكيد إنه ما يُنفَّذ
2. **IDOR** — تعديل رقم ID بطلب شبكي (Network tab) والتأكد ما يرجّع بيانات حساب ثاني
3. **تجاوز Rate Limiting** — محاولات دخول خاطئة متكررة، تأكيد القفل يشتغل
4. **أمان الكوكيز** — F12 → Application → Cookies، تأكيد HttpOnly + Secure مفعّلين
5. **تسريب الأسرار** — بحث بكود المصدر عن `service_role`, `SECRET`, `PRIVATE_KEY`
6. **إساءة استخدام منطق العمل** — تجاوز حد استخدام كود خصم، أو حجز متعارض

⚠️ اختبارات الكتابة (XSS، استغلال الأكواد) على صالون تجريبي فقط، مو "الوادي" الحقيقي.
