0. الاستراتيجية الهجينة (Hybrid Workflow) — أول اتفاق
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

سنعتمد نموذج عمل هجين يتناسب مع موقعك وأدواتك المتاحة:

1. عندما تكون خارج البيت (آيباد):
   - إشارة التحويل: "أنا أعمل الآن من الآيباد خارج البيت"
   - البيئة: Remote Sandbox (السحابية)
   - git commit بعد كل مجموعة تغييرات ثم git push فوراً

2. عندما تكون في البيت (كمبيوتر):
   - إشارة التحويل: "أنا أعمل الآن من جهاز الكمبيوتر في البيت"
   - البيئة: Local Environment (المحلية)
   - تشغيل npm run dev ومعاينة فورية قبل الرفع

3. القاعدة الذهبية للمزامنة:
   - في نهاية أي جلسة: git push
   - في بداية أي جلسة جديدة: git pull

4. نقاط الأمان:
   ✅ بدون تضارب (Conflicts)
   ✅ بدون فقدان أكواد
   ✅ مرونة كاملة من أي مكان


قاعدة مهمة — قبل رفع أي كود:
  1. اختبار محلي أولاً
  2. التأكد من عدم وجود أخطاء
  3. التأكد من أن الميزة تعمل صح
  4. عرض النتائج للمراجعة
  5. فقط بعد موافقتك "OK" يتم الرفع

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

عندما تقول "ارفع"، أتبع هذه الخطوات بالتسلسل:
  1. اختبار محلي — بدون أخطاء
  2. Git commit — برسالة واضحة
  3. Git push — للـ GitHub
  4. شيك الـ PR في GitHub (Merge button أخضر؟)
  5. شيك: All checks passed + No conflicts
  6. اضغط "Merge pull request"
  7. انتظر رسالة البريد من Vercel
  8. أخبرك "التحديث وصل بنجاح" — فقط بعد الـ deployment

✅ ما أقول "وصل" إلا لما كل شيء تمام

  (Vercel error، GitHub conflict، bug في الكود)
✅ أشرح السبب والحل الفوري قبل ما أستمر

19. التزامن اللحظي الثلاثي (Real-time Sync)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

✅ أي ميزة جديدة تمس البيانات لازم تكون لحظية في ثلاثة اتجاهات:
   - عميل ↔ صالون (الحجوزات، التقييمات، الإلغاء)
   - صالون ↔ إدارة (الحجوزات، الإشعارات، الإعدادات)
   - عميل ↔ إدارة (الإشعارات، البيانات)
✅ الأدوات: Supabase Realtime + Polling 10 ثواني + visibilitychange
✅ في حالة الشك → اسأل بدل الافتراض

20. القواعد الذهبية لاستعلامات قاعدة البيانات (NON-NEGOTIABLE)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

الهدف: منع تسرب البيانات مثل حادثة 18GB Egress.

القاعدة 1: ممنوع select('*') — تحديد الأعمدة المطلوبة فقط
القاعدة 2: فلتر إلزامي — أي استعلام لـ bookings/messages/waiting_list
          يجب أن يحتوي على WHERE (salon_id أو customer_id)
القاعدة 3: Limit إلزامي — استخدم limit دائماً:
   - Messages/Chat:  limit=50
   - Reviews:        limit=20
   - Bookings:       limit=100-1000
   - Customers:      limit=500
القاعدة 4: RLS Awareness — افترض الحماية مفعلة دائماً

قائمة التحقق قبل كل Commit:
  [ ] هل جميع الاستعلامات تحدد الأعمدة (بدون *)؟
  [ ] هل جميع الاستعلامات الحساسة لها فلتر WHERE؟
  [ ] هل أضفت limit حيث ينبغي؟
  [ ] هل الاستعلام يحترم RLS؟

21. ميثاق الأداء والوقاية التقنية
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

القاعدة 1 — الاستعلام الجراحي:
  ممنوع select(*) — تحديد الأعمدة فقط دائماً
  كل استعلام يجب أن يحتوي على limit منطقي

القاعدة 2 — مكافحة التكرار:
  ممنوع نفس الـ API في أكثر من useEffect
  أي loadData في مكان واحد مركزي فقط

القاعدة 3 — نظافة الخلفية:
  أي Realtime Channel يجب أن يُغلق بـ unsubscribe()
  Circuit Breaker: توقف بعد 3 محاولات فشل، انتظر 30 ثانية

القاعدة 4 — الاستهلاك الواعي:
  ممنوع تحميل الصور الخام — استخدم Thumbnails/Optimized
  البيانات الثابتة في localStorage (Static Data: 7 أيام)

فحص ذاتي إجباري قبل أي كود جديد:
  [ ] هل استخدمت select=* في أي استعلام؟ (يجب إصلاحه)
  [ ] هل كل استعلام فيه limit؟
  [ ] هل نفس API في أكثر من useEffect؟ (يجب توحيدها)
  [ ] هل كل Realtime subscription لها cleanup؟
  [ ] هل الصور محسّنة وليست خام؟

22. ميثاق نظافة الكود وإدارة التكرار
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

① سياسة النقل الآمن:
  عند نقل أي كود من ملف لآخر → حذفه فوراً من مكانه السابق.
  لا يُسمح بأكواد ميتة (Dead Code) في المشروع.

② الإدارة الدورية للإضافات:
  أي ميزة تُحذف = حذف كامل لجميع الأكواد المرتبطة بها.
  ممنوع ترك أجزاء برمجية معلقة.

③ الوقاية من التضارب:
  حجم الملفات في الحد الأدنى الضروري.
  لا نسخ مكررة من نفس المنطق في أماكن مختلفة.

④ المسؤولية:
  أي كود مكرر أو مهمل = خطأ تقني يستوجب المعالجة الفورية.

23. بروتوكول التكامل البرمجي الصارم (Protocol 23)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. قاعدة الملف الكامل: ممنوع تقديم أجزاء من الكود.
   يجب دائماً إعادة الملف كاملاً (كود قديم + جديد مدمجين).

2. حرمة الكود القائم: ممنوع حذف أي سطر موجود بدون موافقة صريحة.
   الكود الحالي هو المرجع الثابت — أي تعديل إضافة تراكمية فقط.

3. بروتوكول الحذف الشامل:
   عند حذف أي ميزة:
   - مسح جذري: تتبع جميع الدوال والمكونات والمتغيرات وحذفها
   - إزالة التبعيات: Imports غير مستخدمة + مكتبات + Routes
   - تنظيف ملفات الترجمة: حذف المفاتيح من ملفات JSON
   - منع Dead Code: لا كود معلق ولا دوال غير مستخدمة
   - التوثيق: إبلاغ المطور بالإجراءات المتخذة

4. قابلية التنفيذ المباشر: الكود جاهز للنسخ واللصق مباشرة.

5. تنبيه التغيير: لو في تعارض → توقف وأبلغ المستخدم فوراً.

6. التزامن والتحقق من الأسطر:
   - عند إضافة ميزة: عدد الأسطر يجب أن يزيد
   - عند حذف ميزة: عدد الأسطر يجب أن ينقص

24. ميثاق الجراحة البرمجية والمسؤولية الكاملة
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

القاعدة الأساسية: العمل موجّه وجراحي 100%

① قفل النطاق (Scope Lock):
  كل طلب له نطاق محدد — أعمل داخله فقط.
  ما لم يُذكر في الطلب = محظور اللمس.
  قبل أي تعديل: أحدد الملف والسطر المستهدف بدقة.

② المسؤولية الكاملة عن التبعات:
  إذا طُلب مني تعديل A وانكسرت B نتيجة عملي:
  → أنا المسؤول الأول والأخير.
  "لم أقصد" ليست مبرراً.

③ التحقق الإلزامي:
  قبل العمل: أقرأ الكود المحيط وأفهم التبعيات.
  بعد العمل: أتحقق أن ما خارج النطاق لم يتأثر.

④ بروتوكول الشك:
  إذا شعرت أن التعديل قد يمس جزءاً آخر:
  → أتوقف وأخبر المستخدم فوراً.

⑤ الإصلاح الفوري بلا جدل:
  إذا اكتُشف فقدان أي ميزة بعد عملي:
  → أُصلحها فوراً كأولوية قصوى، بلا مبررات.

⑥ إعلان النطاق قبل البدء:
  "سأعدل: [الملف / السطر / المكون]"
  "لن أمس: كل ما عدا ذلك"

⑦ تقرير التغيير بعد الإنجاز:
  - ماذا عدّلت بالضبط (الملف + السطر)
  - ماذا تركت دون تغيير
  - هل يوجد أي خطر جانبي محتمل

دورة العمل: إعلان النطاق ← تنفيذ جراحي ← تحقق ← تقرير

25. ميثاق الصدق في الإنجاز وانضباط الرفع
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

القاعدة: كل "تم" كاذبة = خيانة لوقت المستخدم وتوكنزاته.

① يُحظر إعلان الإنجاز قبل التحقق الفعلي:
  "تم" لا تُكتب إلا بعد قراءة الكود المُعدَّل والتأكد البصري.

② الصراحة التامة واجبة:
  إذا التحقق يحتاج تشغيلاً من المستخدم → أقول ذلك صراحةً.

③ مرتان تفشل = توقف كامل وتغيير النهج:
  لا محاولة ثالثة بنفس الطريقة.
  تشخيص السبب الجذري + طلب معلومة ناقصة.

④ ترتيب الرفع إلزامي:
  تحقق ← دمج ← رفع ← إصدار
  من يقول "رفعت" دون دمج = لم يرفع.

⑤ رقم الإصدار إلزامي بعد كل رفع ناجح:
  الصيغة: حرف إنجليزي + أرقام (مثال: L46)
  الرقم يزيد تسلسلياً مع كل إصدار.

⑥ كل commit يجب أن يكون حقيقياً ومتحققاً.
  سلسلة commits بدون تحقق = استنزاف للـ rate limit.

⑦ تكلفة الوعد الكاذب يدفعها كلود — لا المستخدم.
