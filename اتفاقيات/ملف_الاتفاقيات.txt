0. الاستراتيجية الهجينة — أول اتفاق

عندما تكون خارج البيت (آيباد):
- قل "أنا أعمل الآن من الآيباد خارج البيت"
- البيئة: Remote Sandbox
- git commit ثم git push فوراً بعد كل تغييرات

عندما تكون في البيت (كمبيوتر):
- قل "أنا أعمل الآن من جهاز الكمبيوتر في البيت"
- البيئة: Local Environment
- تشغيل npm run dev ومعاينة فورية قبل الرفع

القاعدة الذهبية للمزامنة:
- نهاية أي جلسة: git push
- بداية أي جلسة جديدة: git pull

قاعدة مهمة قبل رفع أي كود:
1. اختبار محلي أولاً
2. التأكد من عدم وجود أخطاء
3. التأكد من أن الميزة تعمل صح
4. عرض النتائج للمراجعة
5. فقط بعد موافقتك "OK" يتم الرفع


================================================================
الاتفاقيات الأساسية
================================================================


1. طريقة العمل

خطأ: تعطيني مهمة وأنا أبدأ بسرعة بدون استشارة
الصحيح: خطوة بخطوة، استشارة قبل التنفيذ، بدون تسرع
كل خطوة أعرضها عليك قبل ما أكملها


2. قبل أي تعديل

خطأ: أعدل الكود مباشرة بدون backup أو خطة
الصحيح: أخذ نسخة قديمة قبل أي تعديل
عرض الخطة أولاً وانتظار الموافقة "OK" قبل التنفيذ


3. الرفع للـ GitHub

خطأ: رفع الكود بدون موافقة أو force push بدون إذن
الصحيح: الرفع بعد الموافقة فقط
مع رسالة commit واضحة ومفصلة
بدون force push بدون إذن صريح


4. الاختبار

خطأ: رفع الكود بدون اختبار محلي
الصحيح: التأكد من عدم وجود أخطاء قبل الرفع
شرح النتائج بوضوح للمستخدم


5. اتفاقية الترابط والإدارة

خطأ: إضافة ميزة للعميل/الصالون بدون تفعيلها في لوحة الإدارة
القاعدة الذهبية: أي إضافة مهمة (PIN، بريد، رقم جوال)
يجب تكون في Admin Panel مع 2-way sync
الإضافات المهمة أضيفها في التطبيق والإدارة معاً بدون سؤال
الإضافات البسيطة (ألوان، خطوط) أسأل أولاً
في حالة الشك: اسأل بدل الافتراض


6. الأمان والبيانات الحساسة

خطأ: حذف بيانات العملاء بدون إذن
الصحيح: بدون حذف بيانات العملاء بدون إذن صريح
حماية البيانات الشخصية والحساسة
التعامل الآمن مع كلمات المرور والـ PIN


7. الاختبار على جميع الأجهزة

خطأ: رفع ميزة بدون اختبار على جميع الأجهزة
الصحيح: اختبار على iPhone وiPad وAndroid والكمبيوتر
بدون رفع حتى نتأكد من النتائج على الكل


8. توثيق التغييرات

خطأ: رسالة commit غير واضحة مثل "fix" أو "update"
الصحيح: رسالة commit واضحة ومفصلة
شرح السبب وليس فقط الماذا


9. سجل النسخ الاحتياطية

خطأ: حذف الملفات القديمة بعد التعديل
الصحيح: احتفظ بنسخة قديمة من أي تعديل مهم
لا تحذف الملفات القديمة بدون سبب


10. التوافقية والأداء

خطأ: إضافة ميزة تبطي التطبيق أو تكسر التوافقية
الصحيح: بدون إبطاء التطبيق
التأكد من التوافقية مع جميع الإصدارات


11. خطوات النشر الموحدة

عندما تقول "ارفع"، أتبع هذه الخطوات بالتسلسل:
1. اختبار محلي بدون أخطاء
2. Git commit برسالة واضحة
3. Git push للـ GitHub
4. شيك الـ PR في GitHub
5. التأكد من: All checks passed وNo conflicts
6. اضغط Merge pull request
7. انتظر رسالة البريد من Vercel
8. أخبرك "التحديث وصل بنجاح" فقط بعد الـ deployment
ما أقول "وصل" إلا لما كل شيء تمام


12. التواصل في الحالات الطارئة

خطأ: الاستمرار رغم وجود مشكلة غير متوقعة
الصحيح: لو حصلت مشكلة غير متوقعة أخبرك فوراً
أشرح السبب والحل الفوري قبل ما أستمر


13. تسهيل سير العمل

أعطني السياق الكامل في البداية
ثق بالعملية — لو بدأت شيء خليني أكمله
لا تقاطع متكرر
ملاحظات واضحة من البداية


14. التركيز على المطلوب فقط

خطأ: مهمة واحدة فأشتغل على أشياء لم تطلبها
الصحيح: أشتغل على بالضبط اللي طلبته بس
لو رأيت تحسينات أخرى أسألك أولاً


15. اسأل عند عدم الفهم

خطأ: أشتغل على افتراض غير واضح
الصحيح: بدون تخمين، اسأل فوراً عند أي نقطة غير واضحة
أي نقطة غير واضحة = أسأل فوراً، ما أشتغل على افتراضات


16. أنت اللي تسأل

أنا أسأل أسئلة استباقية لأفهم احتياجاتك
ما أنتظر تشرح كل شيء، أنا اللي أسأل


17. اللغة العربية

خطأ: الرد أو الملخصات بالإنجليزي
الصحيح: جميع الردود والملخصات والشرح بالعربية
الكود والأوامر التقنية تبقى بالإنجليزي لكن الشرح عربي


18. معيار التطبيقات الكبرى

مع كل ميزة جديدة، أشرح لك:
- كيف كنا نسويها (طريقتنا الحالية)
- كيف تسويها التطبيقات الكبرى (Airbnb, Talabat, Booking)
- ما الفرق وليش طريقتهم أفضل
الهدف: نكشف الأخطاء الكبيرة الآن قبل ما تصير
أنت تقرر بعد الشرح


19. التزامن اللحظي الثلاثي

أي ميزة جديدة تمس البيانات لازم تكون لحظية في ثلاثة اتجاهات:
- عميل وصالون (الحجوزات، التقييمات، الإلغاء)
- صالون وإدارة (الحجوزات، الإشعارات، الإعدادات)
- عميل وإدارة (الإشعارات، البيانات)
الأدوات: Supabase Realtime مع Polling 10 ثواني

20. القواعد الذهبية لاستعلامات قاعدة البيانات
================================================================

الهدف: منع تسرب البيانات مثل حادثة 18GB Egress

القاعدة الأولى: ممنوع select * — تحديد الأعمدة المطلوبة فقط

القاعدة الثانية: فلتر إلزامي
أي استعلام لجداول bookings أو messages أو waiting_list
يجب أن يحتوي على WHERE مع salon_id أو customer_id

القاعدة الثالثة: Limit إلزامي
- Messages/Chat:  50 حد أقصى
- Reviews:        20 حد أقصى
- Bookings:       100 إلى 1000
- Customers:      500 حد أقصى

القاعدة الرابعة: RLS Awareness
افترض أن Row Level Security مفعلة على جميع الجداول

قائمة التحقق قبل كل Commit:
- هل جميع الاستعلامات تحدد الأعمدة بدون *؟
- هل جميع الاستعلامات الحساسة لها فلتر WHERE؟
- هل أضفت limit حيث ينبغي؟
- هل الاستعلام يحترم RLS؟


================================================================
21. ميثاق الأداء والوقاية التقنية
================================================================

القاعدة الأولى — الاستعلام الجراحي:
ممنوع select * — تحديد الأعمدة فقط دائماً
كل استعلام يجب أن يحتوي على limit منطقي

القاعدة الثانية — مكافحة التكرار:
ممنوع نفس الـ API في أكثر من useEffect
أي loadData في مكان واحد مركزي فقط

القاعدة الثالثة — نظافة الخلفية:
أي Realtime Channel يجب أن يُغلق بـ unsubscribe
توقف بعد 3 محاولات فشل، انتظر 30 ثانية

القاعدة الرابعة — الاستهلاك الواعي:
ممنوع تحميل الصور الخام، استخدم Thumbnails
البيانات الثابتة في localStorage لمدة 7 أيام

فحص ذاتي إجباري قبل أي كود جديد:
- هل استخدمت select=* في أي استعلام؟ يجب إصلاحه
- هل كل استعلام فيه limit؟
- هل نفس API في أكثر من useEffect؟ يجب توحيدها
- هل كل Realtime subscription لها cleanup؟
- هل الصور محسّنة وليست خام؟

22. ميثاق نظافة الكود وإدارة التكرار
================================================================

سياسة النقل الآمن:
عند نقل أي كود من ملف لآخر، حذفه فوراً من مكانه السابق
لا يُسمح بأكواد ميتة في المشروع

الإدارة الدورية للإضافات:
أي ميزة تُحذف = حذف كامل لجميع الأكواد المرتبطة بها
ممنوع ترك أجزاء برمجية معلقة

الوقاية من التضارب:
حجم الملفات في الحد الأدنى الضروري
لا نسخ مكررة من نفس المنطق في أماكن مختلفة

23. بروتوكول التكامل البرمجي الصارم
================================================================

1. قاعدة الملف الكامل:
ممنوع تقديم أجزاء من الكود
يجب دائماً إعادة الملف كاملاً

2. حرمة الكود القائم:
ممنوع حذف أي سطر موجود بدون موافقة صريحة
أي تعديل يجب أن يكون إضافة تراكمية فقط

3. بروتوكول الحذف الشامل:
عند حذف أي ميزة:
- مسح جذري لجميع الدوال والمكونات والمتغيرات
- إزالة Imports غير مستخدمة ومكتبات وRoutes
- تنظيف ملفات الترجمة وحذف مفاتيح JSON
- منع Dead Code — لا كود معلق ولا دوال غير مستخدمة
- إبلاغ المطور بالإجراءات المتخذة

4. قابلية التنفيذ المباشر:
الكود جاهز للنسخ واللصق مباشرة

5. تنبيه التغيير:
لو في تعارض، توقف وأبلغ المستخدم فوراً

6. التزامن والتحقق من الأسطر:
- عند إضافة ميزة: عدد الأسطر يجب أن يزيد
- عند حذف ميزة: عدد الأسطر يجب أن ينقص

24. ميثاق الجراحة البرمجية والمسؤولية الكاملة
================================================================

القاعدة الأساسية: العمل موجّه وجراحي 100%

1. قفل النطاق:
كل طلب له نطاق محدد، أعمل داخله فقط
ما لم يُذكر في الطلب = محظور اللمس
قبل أي تعديل: أحدد الملف والسطر المستهدف بدقة

2. المسؤولية الكاملة عن التبعات:
إذا طُلب مني تعديل نقطة A وانكسرت نقطة B
أنا المسؤول الأول والأخير، بلا استثناء
"لم أقصد" ليست مبرراً

3. التحقق الإلزامي:
قبل العمل: أقرأ الكود المحيط وأفهم التبعيات
بعد العمل: أتحقق أن ما خارج النطاق لم يتأثر

4. بروتوكول الشك:
إذا شعرت أن التعديل قد يمس جزءاً آخر
أتوقف وأخبر المستخدم فوراً

5. الإصلاح الفوري بلا جدل:
إذا اكتُشف فقدان أي ميزة بعد عملي
أُصلحها فوراً كأولوية قصوى، بلا مبررات

6. إعلان النطاق قبل البدء:
"سأعدل: الملف والسطر والمكون"
"لن أمس: كل ما عدا ذلك"

7. تقرير التغيير بعد الإنجاز:
- ماذا عدّلت بالضبط (الملف والسطر)
- ماذا تركت دون تغيير
- هل يوجد أي خطر جانبي محتمل

دورة العمل: إعلان النطاق ← تنفيذ جراحي ← تحقق ← تقرير

25. ميثاق الصدق في الإنجاز وانضباط الرفع
================================================================

القاعدة: كل "تم" كاذبة = خيانة لوقت المستخدم وتوكنزاته

1. يُحظر إعلان الإنجاز قبل التحقق الفعلي:
"تم" لا تُكتب إلا بعد قراءة الكود المُعدَّل والتأكد البصري

2. الصراحة التامة واجبة:
إذا التحقق يحتاج تشغيلاً من المستخدم، أقول ذلك صراحةً

3. مرتان تفشل = توقف كامل وتغيير النهج:
لا محاولة ثالثة بنفس الطريقة
تشخيص السبب الجذري وطلب معلومة ناقصة

4. ترتيب الرفع إلزامي:
تحقق ← دمج ← رفع ← إصدار
من يقول "رفعت" دون دمج = لم يرفع

5. رقم الإصدار إلزامي بعد كل رفع ناجح:
الصيغة: حرف إنجليزي ثم أرقام (مثال: L46)
الرقم يزيد تسلسلياً مع كل إصدار

6. كل commit يجب أن يكون حقيقياً ومتحققاً
سلسلة commits بدون تحقق = استنزاف للـ rate limit

7. تكلفة الوعد الكاذب يدفعها كلود لا المستخدم
كل "تم" كاذبة تُكلّف وقت المستخدم وجهده وتوكنزاته
